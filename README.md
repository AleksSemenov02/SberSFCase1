# SberSFCase1

REST API для автоматизированной проверки комплектности пакетов документов, подаваемых
для оплаты по программам льготного кредитования (`federal` / `regional`).

Сервис принимает пакет файлов, определяет тип каждого документа по имени файла, проверяет
комплектность пакета относительно требований выбранной программы и сохраняет результат
проверки (статус, замечания, список документов) в PostgreSQL. Файлы документов сохраняются
в S3-совместимое хранилище (MinIO).

## Как запустить проект

Проект поднимается одной командой через Docker Compose.

1. Запустите все сервисы:
   ```bash
   docker compose -f contrib/docker/docker-compose.local.yaml up
   ```

   Compose поднимет:
   - `db` — PostgreSQL,
   - `minio` — S3-совместимое хранилище для файлов,
   - `migrations` — применяет миграции Alembic перед стартом API,
   - `app_api` — сам сервис, доступен на `http://localhost:8000`.

2. Swagger-документация доступна по адресу:
   ```
   http://localhost:8000/docs
   ```

3. Консоль MinIO (для просмотра загруженных файлов) — `http://localhost:9001`
   (логин/пароль — значения `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` из `.env.example`).

## Как запустить тесты

Тесты выполняются в отдельном сервисе `tests`, использующем тот же образ, что и `app_api`:

```bash
docker compose -f contrib/docker/docker-compose.local.yaml run tests
```

Тесты покрывают ключевую бизнес-логику модуля проверки документов (`app/api/checks/helpers.py`):
- определение типа документа по имени файла (`detect_doc_type`);
- формирование итогового статуса проверки на основе комплектности пакета
  (`check_all_doc_types_for_program`).

## Технологии

| Технология | Назначение |
|---|---|
| **FastAPI** | REST API, автогенерация OpenAPI-документации, встроенная валидация через Pydantic |
| **SQLAlchemy 2.0 (async)** | ORM, работа с PostgreSQL в асинхронном режиме |
| **Alembic** | миграции схемы БД |
| **PostgreSQL** | хранение метаданных проверок, документов и замечаний, история изменений |
| **asyncpg** | асинхронный драйвер PostgreSQL для SQLAlchemy |
| **aioboto3** | асинхронный клиент для загрузки файлов в S3-совместимое хранилище (MinIO) |
| **MinIO** | S3-совместимое файловое хранилище (локальная замена облачного S3 для разработки) |
| **fastapi-pagination** | пагинация списка проверок в `GET /api/checks` |
| **pydantic-settings** | конфигурация приложения через переменные окружения |
| **pytest** | тестирование бизнес-логики |

Стек выбран согласно требованиям задания (Python 3.11, FastAPI, PostgreSQL, SQLAlchemy) и
дополнен стандартными для этого стека инструментами: `aioboto3` — для асинхронной работы с
файловым хранилищем без блокировки event loop, `fastapi-pagination` — чтобы не писать пагинацию
вручную, `pydantic-settings` — для типобезопасной конфигурации через `.env`.

## Архитектура решения

```
app/
├── api/
│   ├── checks/
│   │   ├── views.py      — эндпоинты /api/checks
│   │   ├── helpers.py    — бизнес-логика: классификация документов,
│   │   │                    проверка комплектности пакета, сохранение файлов в S3
│   │   └── schemas.py    — Pydantic-схемы запросов/ответов, enum'ы (DocType, ProgramType, Status)
│   ├── responses.py      — кастомные HTTP-исключения (400/404)
│   └── schemas.py        — общие схемы ответов об ошибках
database/
├── models.py              — SQLAlchemy-модели (CheckModel, DocumentModel, IssueModel)
└── migrations/            — миграции Alembic
s3/
└── helpers.py              — клиент для работы с S3/MinIO
contrib/
└── docker/
    ├── app_api/Dockerfile
    └── docker-compose.local.yaml
tests/
└── api/checks/test_helpers.py — тесты бизнес-логики
```

**Модель данных.** Одна проверка (`CheckModel`) — это пакет документов, поданный за один
запрос. У проверки есть уникальный `id`, программа (`federal`/`regional`), итоговый статус и
дата. К проверке привязаны:
- `DocumentModel` — распознанные документы пакета (имя, тип, размер);
- `IssueModel` — список замечаний (`error` — блокирует одобрение, `warning` — не блокирует).

Такая связка `CheckModel → DocumentModel / IssueModel` даёт полную историю: по `id` проверки
всегда можно поднять, что именно было загружено и почему проверка получила тот или иной статус,
без необходимости хранить сырые файлы где-то ещё, кроме S3.

**Поток обработки запроса `POST /api/checks`:**
1. Валидация входных данных через Pydantic (`program_type`, `files`) — некорректный запрос
   отклоняется на уровне FastAPI с кодом 422.
2. Для каждого файла: проверка расширения (`pdf`, `docx`, `jpg`, `png`) и размера (≤ 20 МБ),
   определение типа документа по паттернам в имени файла.
3. Файлы, прошедшие проверку, загружаются в S3 (MinIO); для остальных формируется `warning`.
4. Сравнение набора распознанных типов документов с обязательным набором для выбранной
   программы — формируется список `error`-замечаний по недостающим документам и итоговый статус
   (`approved` / `rejected`).
5. Результат (проверка, документы, замечания) сохраняется в PostgreSQL одной транзакцией.

**Эндпоинты:**
- `POST /api/checks/` — принимает `multipart/form-data` (файлы + `program_type`), запускает
  проверку, возвращает и сохраняет результат.
- `GET /api/checks/` — постраничный список всех проверок.
- `GET /api/checks/{id}/` — полный результат конкретной проверки (404, если не найдена).

## Переменные окружения

Описание переменных из `.env.example`:

| Переменная | Описание |
|---|---|
| `AWS_ACCESS_KEY_ID` | ключ доступа к S3-совместимому хранилищу (в локальной разработке — логин MinIO) |
| `AWS_SECRET_ACCESS_KEY` | секретный ключ доступа к хранилищу (в локальной разработке — пароль MinIO) |
| `BUCKET_NAME` | имя бакета, в который сохраняются загруженные документы |
| `DATABASE_URL` | строка подключения к PostgreSQL в формате `postgresql+asyncpg://<user>:<password>@<host>/<db>` |
| `APP_DEBUG` | флаг режима отладки FastAPI (`True`/`False`) |

> В `docker-compose.local.yaml` MinIO и PostgreSQL поднимаются с теми же учётными данными,
> что указаны в `.env.example` — для локального запуска менять их не обязательно.
